# 小红书 AI 大模型事件传播采集计划

## 1. 研究边界

本项目只研究“中外主流大模型发布/升级事件在小红书的传播”，不把所有 AI 工具热词都混入核心样本。核心事件必须属于同一类型：模型发布、模型升级、模型可用性/价格/能力重大变更。

研究窗口采用两层定义：

- 平台采集窗口：2026-04-01 至 2026-07-04。4 月初数据接近 90 天边界，优先尽快采集。
- 事件分析窗口：事件日前 3-7 天至事件日后 7-30 天；必要时保留长尾至 2026-07-04。

对比主题如“国产大模型 vs OpenAI / Claude”不作为单独核心事件，而作为核心事件采完后的比较维度。

## 2. 入库标准

候选事件采集后，只有达到以下最低标准，才进入核心分析样本：

| 指标 | 最低建议 |
|---|---:|
| 相关笔记数 | >= 50 条 |
| 有效评论数 | >= 500 条 |
| 有地区信息的评论 | >= 200 条 |
| 参与省份/城市 | >= 15 个 |
| 时间跨度 | 至少连续 7 天有讨论 |
| 非营销内容比例 | >= 60% |

分级规则：

- 核心事件：全部或基本满足以上阈值，可进入主模型。
- 观察事件：笔记不少于 30 条，但评论、地区或时间跨度不足，保留作补充案例。
- 剔除事件：主要是广告、课程、工具泛词，或无法定位到明确模型发布/更新事件。

## 5. 采集流程

### 阶段 A：候选事件侦察

目标：判断 10-12 个候选事件是否有足够数据量。

做法：

- 只抓笔记，不抓评论。
- 每个事件 6-8 个关键词。
- 每个关键词先抓 20 条左右。
- 去重后统计相关笔记数、日期跨度、营销比例、互动量。

输出：

- `event_candidate_status.csv`
- `notes_raw_merged.csv`
- `event_screening_summary.csv`

进入下一阶段的最低条件：

- 去重后相关笔记 >= 50，或
- 去重后相关笔记 >= 30 且评论总量潜力明显。

### 阶段 B：评论试采

目标：判断核心候选是否满足评论和地区阈值。

做法：

- 每个候选事件选 15-20 条高价值笔记。
- 每条笔记抓 30 条一级评论。
- 暂不抓二级评论。

预期：

- 每事件 450-600 条评论。
- 有地区信息评论应不少于 200。
- 涉及地区应不少于 15 个。

输出：

- `comments_pilot.csv`
- `event_core_threshold_check.csv`

### 阶段 C：核心事件深采

目标：形成最终分析数据。

做法：

- 选择 5-8 个核心事件。
- 每事件保留 50-300 条相关笔记。
- 每事件抓 500-3000 条评论。
- 对争议高、评论多、传播长尾明显的事件，可补抓二级评论。

最终规模：

- 核心事件：5-8 个。
- 笔记：约 500-1500 条。
- 评论：约 5000-15000 条。

这个规模适合硕士/项目研究，既有统计稳定性，也不会失控。

## 7. 内容判定规则

### 7.1 相关笔记

满足以下任一条件：

- 标题、正文或话题中出现事件名或模型名。
- 明确讨论该模型的发布、上线、更新、测评、价格、API、教程、能力变化。
- 明确将该模型与 OpenAI、Claude、国产模型等进行对比。

### 7.2 非相关笔记

剔除：

- 只出现品牌词，但没有讨论该事件。
- 纯 AI 泛教程，与事件无关。
- 与模型无关的账号、壁纸、娱乐、无效广告。

### 7.3 营销内容

标记为营销，不一定删除：

- 包含“私信领取”“课程”“训练营”“代开”“账号”“资料包”“群”“代理”等强推广词。
- 主要目的不是讨论事件，而是引流或售卖服务。

核心事件要求非营销内容比例 >= 60%。

## 9. Agent 模拟采样方案

本项目最终要进入 agent 模拟，因此采集对象需要分成“行为事件”和“匿名行为体画像”两类。核心原则是：

> 总量采集“行为事件”，有限采集“行为体画像”，不采集“真人链条身份”。

### 9.1 发帖作者采样

采样对象：

- 核心事件下所有相关笔记作者。

采集内容：

- 主页公开统计。
- 最近 20 条公开作品。
- 如果作者数量较少，可扩展到最近 50 条公开作品。

使用目的：

- 判断发帖者是官方账号、媒体账号、技术博主、普通用户还是营销账号。
- 评估发帖者历史 AI/科技内容比例。
- 构建发布者影响力分层，而不是还原真人身份。

### 9.2 评论基础行为采样

采样对象：

- 核心事件下所有评论。

采集内容：

- 评论文本。
- 评论时间。
- 评论地区。
- 评论点赞数。
- 评论者匿名哈希。
- 是否为回复。
- 父评论 ID。

使用目的：

- 构建事件传播的时间序列、地域扩散、意见类型和互动强度。

### 9.3 评论者主页画像采样

不对所有评论者追踪主页，只对以下分层样本采集公开画像：

1. 每个事件评论点赞数 Top 5%。
2. 每个事件最早 24 小时评论者。
3. 跨 2 个及以上事件出现的评论者。
4. 每个重点地区随机抽样评论者。
5. 知识型、争议型、风险型评论者。

采集内容：

- 主页是否可访问。
- 公开粉丝数区间。
- 公开关注数区间。
- 公开获赞收藏数区间。
- 公开 IP 地区。
- 公开作品抽样数量。
- AI/科技相关作品比例。
- 推断账号类型。

### 9.4 不采集内容

以下字段不进入最终研究数据集：

- 原始 `user_id`
- `redId`
- 主页 URL
- 头像 URL
- 原始昵称
- `xsec_token`
- 关注列表
- 粉丝列表
- 点赞列表
- 私密或半私密内容

如果采集过程中接口需要临时使用 `user_id` 或 `xsec_token`，只允许在内存中用于请求，不写入最终 CSV。

### 9.5 总体采集策略表

| 体系 | 采集策略 |
|---|---|
| 笔记 | 全部量 |
| 评论 | 全部量 |
| 发帖作者公开统计 | 核心作者全量 |
| 发帖作者历史公开作品 | 20-50 条有限采样 |
| 评论者基础行为 | 全部量 |
| 评论者主页画像 | 分层结构 |
| 评论者关系链 / 点赞链 / 身份标识 | 不采 |

## 10. 多设备采集方案

可以使用 3 台电脑、3 个账号，但需要统一代码和输出结构。

### 10.1 设备分工

| 设备 | 账号 | 事件组 |
|---|---|---|
| Device A | Account A | DeepSeek / GPT |
| Device B | Account B | Claude / Gemini |
| Device C | Account C | Qwen / 豆包 / Kimi / MiniMax / 图像视频模型 |

### 10.2 运行原则

- 每台设备单独 Chrome profile。
- 每批只跑 3-5 个主题。
- 每批间隔 30-60 分钟。
- 评论深采批次间隔 1-2 小时。
- 触发验证码立即停止，冷却 3-6 小时。
- 不追求并发，追求稳定覆盖。

### 10.3 目录命名

每台设备统一：

```text
runs/xhs_core_events/
  device_a/
    notes/
    comments/
  device_b/
    notes/
    comments/
  device_c/
    notes/
    comments/
  merged/
```

合并时用 `event_id + note_id + comment_id` 去重。

## 11. 时间预估

在 3 台设备、3 个账号、低频采集策略下：

| 阶段 | 目标 | 预计时间 |
|---|---|---:|
| 事件侦察 | 10-12 个候选事件笔记层 | 0.5-1 天 |
| 阈值判断 | 清洗、去重、筛核心候选 | 1-2 小时 |
| 评论试采 | 每候选 450-600 条评论 | 1-2 天 |
| 核心深采 | 5-8 个事件、5000-15000 评论 | 2-4 天 |
| 最终清洗 | 合并、去重、质量表 | 0.5 天 |

总计：

- 快速版本：2-3 天。
- 稳妥版本：4-6 天。
- 若频繁触发验证码：7 天左右。

## 12. 当前执行建议

基于当前已有数据：

- `DeepSeek-V4` 已经满足继续深采条件。
- `DeepSeek V4 Pro`、`DeepSeek V4 Flash`、`DeepSeek 1M上下文` 已有笔记基础，可补评论。
- `DeepSeek Claude Code`、`GPT-5.5` 为低覆盖，需要补笔记。
- 其余候选事件需要重新分批跑。

下一步不应再一次性跑 20+ 主题，应改为每次 3 个主题，并严格记录采集批次、设备、账号和触发风控情况。
